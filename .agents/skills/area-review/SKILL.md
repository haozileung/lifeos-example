# 领域回顾 (Area Review)

对 LifeOS 中的领域进行系统性回顾和评分，帮助用户了解各领域的健康状况并制定改进计划。

## 触发条件

当用户提到以下内容时触发：
- 领域回顾、领域复盘、领域评分、领域健康检查
- "帮我回顾一下XX领域"、"XX领域这个季度怎么样"
- "做个季度复盘"、"area review"
- 也可以在用户未指定领域时，主动建议对特定领域做回顾

## 工作流程

### Step 1: 确认参数

向用户确认以下参数（如果用户已提供则跳过）：

1. **领域**: 要回顾哪个领域？默认列出 `2.Areas/` 下的所有活跃领域供选择
2. **时间范围**: 默认为当前季度（按自然季度计算），也支持月度、年度或自定义日期范围
3. **回顾模式**:
   - **单领域回顾**: 深度分析一个领域
   - **全局回顾**: 对所有活跃领域做快速扫描和对比

如果用户说"做个领域回顾"但没指定具体领域，列出活跃领域并询问。

### Step 2: 采集数据

运行数据采集脚本：

```bash
python3 .claude/skills/area-review/scripts/collect_area_data.py --area <area-folder-name> --period <period>
```

参数说明：
- `--area`: 领域文件夹名（如 `personal-health`、`career-development`），或 `all` 做全局回顾
- `--period`: `current-quarter`（默认）、`last-quarter`、`current-month`、`last-month`、`2026-Q1`、`2026-01`、`2026`，或自定义 `2026-01-01:2026-03-31`

脚本输出：
- 领域 README 的关键信息（维护标准、成功标准、指标）
- 时间范围内的相关任务统计（已完成 / 未完成）
- 时间范围内的笔记提及次数
- 相关项目的进展状态

### Step 3: 分析与评分

读取脚本输出的数据和领域 README，按四个维度进行评分。评分原则：**基于证据，不是主观猜测**。每个评分必须有对应的数据支撑。

**重要：数据局限性提醒**
- 脚本通过标签匹配采集数据，只统计了带标签的任务和 bullet
- 用户可能在日记中以长文、wiki-link 引用、嵌入内容等形式记录了大量领域相关信息，这些不会被脚本捕获
- `tags_actually_matched` 字段显示实际匹配到的标签，可能与 README 中定义的 topic tags 不同（因为使用了前缀匹配）
- `engagement_ratio`（活跃天数/总天数）低不一定代表不关注，可能只是没有用标签记录
- 在评分时，如果脚本数据明显不足，应**额外阅读几篇日记**来补充上下文，而不是仅依赖脚本数据

#### 四维评分框架

**维度一：维护达标率 (Maintenance Score)**

评估该领域的日常维护任务完成情况。

- 数据来源：领域 README 中的维护任务 + 周期笔记中标记了领域标签的任务
- 评分标准：
  - **9-10**: 绝大多数维护任务按计划完成，维护频率稳定
  - **7-8**: 大部分维护任务完成，偶有遗漏但及时补救
  - **5-6**: 约一半维护任务完成，有明显 gaps
  - **3-4**: 维护严重不足，多项标准未达标
  - **1-2**: 几乎没有维护记录

**维度二：项目推进 (Project Progress)**

评估该领域关联的改善项目进展情况。

- 数据来源：领域 README 中列出的相关项目 + 项目 README 的进展状态
- 评分标准：
  - **9-10**: 关键项目按计划推进或已完成，里程碑达成
  - **7-8**: 项目整体在正轨上，有可控的延迟
  - **5-6**: 部分项目推进中，部分停滞
  - **3-4**: 多数项目进展缓慢或方向不明
  - **1-2**: 关键项目未启动或严重脱轨

**维度三：资源投入 (Resource Investment)**

评估在该领域投入的时间和注意力。

- 数据来源：周期笔记中标记了领域相关标签的任务数量、bullet 数量、提及频率
- 评分标准：
  - **9-10**: 持续稳定的关注和记录，投入充沛
  - **7-8**: 有规律的关注，偶有低谷
  - **5-6**: 投入不均衡，时多时少
  - **3-4**: 关注明显不足，记录稀疏
  - **1-2**: 几乎没有投入记录

**维度四：综合健康度 (Overall Health)**

综合前三维度，结合领域 README 中的成功标准和指标进行整体评估。

- 这是一个**加权综合评分**，不是前三项的简单平均
- 需要考虑：
  - 领域 README 中定义的成功标准达成情况
  - 领域特有的指标（如 career-development 中的"管理者行为占比"）
  - 领域当前阶段和战略重要性
  - 趋势判断：是在改善还是恶化？

#### 全局回顾报告格式

保存到 `0.PeriodicNotes/<year>/Quarterly/<year>-Q<q>-area-review.md`。

包含：
1. 所有领域的评分对比表
2. 各领域一句话总结

### Step 5: 呈现与确认

1. 在对话中展示报告的核心内容（评分总览 + 关键发现 + 建议）
2. 询问用户是否满意，是否需要调整评分或补充内容
3. 用户确认后写入文件

## LifeOS 集成要点

### 数据路径
- 领域 README: `2.Areas/<area-folder>/<area-folder>.README.md`
- 周期笔记: `0.PeriodicNotes/<year>/Daily/<MM>/<yyyy-MM-dd>.md`
- 周记: `0.PeriodicNotes/<year>/Weekly/<yyyy-Www>.md`
- 月记: `0.PeriodicNotes/<year>/Monthly/<yyyy-MM>.md`
- 季度笔记: `0.PeriodicNotes/<year>/Quarterly/<yyyy-Qq>.md`
- 项目 README: `1.Projects/<project-folder>/<project-folder>.README.md`

### 标签匹配
- 领域 README 的 `tags` 中包含 topic tag（如 `personal/health`、`career/development`）
- 用这些 topic tag 搜索周期笔记中的相关任务和 bullet
- 任务行格式：`- [x] 完成训练 #personal/health`

### 时间范围计算
- 自然季度：Q1=1-3月，Q2=4-6月，Q3=7-9月，Q4=10-12月
- ISO 周数：`W01`-`W53`
- 日期格式：`YYYY-MM-DD`

### 财年支持
- Deckers 财年：FY 从4月开始，Q1=4-6月，Q2=7-9月，Q3=10-12月，Q4=1-3月
- 如果领域 README 使用财年标签，报告中同时标注自然季度和财年

## 评分注意事项

1. **诚实评分**: 不要因为数据不足就给高分。数据不足本身就是一个信号。
2. **区分"没记录"和"没做"**: 如果用户可能做了但没记录，在评价中标注"数据缺失"，但评分应反映记录的完整度。
3. **趋势比绝对值重要**: 关注变化方向，而不只是当前分数。
4. **加权考虑**: 对用户当前战略重点的领域，综合健康度应更严格。
5. **具体可行**: 建议必须具体、可执行、与证据直接相关。避免泛泛的"继续努力"。
6. **中文输出**: 所有报告内容使用中文，保留英文术语（如 PARA、LifeOS）作为专有名词。
